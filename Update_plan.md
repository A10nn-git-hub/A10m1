# БЛОК 1
## 🛠️ Архитектурный план разработки: GEMINI ANTIGRAVITY
## Распределение задач по специализированным ИИ-агентам

Этот документ определяет технические требования для оптимизации 2D-шутера в Telegram-боте. Каждая задача должна обрабатываться строго закрепленным за ней ИИ-агентом для обеспечения максимальной чистоты кода и изоляции контекста.

--- 

### 1. 📋 Оптимизация игрового статуса (Rich Presence)
* **Ответственный исполнитель:** `[Агент: UI/UX и Системный Статус]`
* **Текущая проблема:** Статус игрока в режиме командного боя перегружен лишней информацией («играет в командный бой 5х5»).
* **Инструкция для Агента:**
    * Для режима **Командный бой** принудительно сократить строку до: `«Играет в командный бой»` (полностью вырезать формат матча *5х5*).
    * Для остальных игровых режимов (например, **Дуэли**) сохранить текущую логику с выводом формата или количества участников (например, *1х1*, *2х2*) для обеспечения информативности.

---

### 2. 🎛️ Шаблоны (пресеты) конфигурации лобби
* **Ответственный исполнитель:** `[Агент: Логика Управления и Интерфейсы]`
* **Суть изменения:** Внедрение трех фиксированных глобальных пресетов настроек, исключающих ручную рутину при создании матча. Игроки не могут их редактировать, но применяют в один клик.
* **Инструкция для Агента:**
    * Спроектировать промежуточный интерфейс кастомных настроек в виде вертикального стека из трех кнопок (в столбик): `ПОБЕДНЫЙ`, `СКОРОСТНОЙ`, `ПОЛЬЗОВАТЕЛЬСКИЕ`.
    * **Режим №1 (ПОБЕДНЫЙ):** Выставить параметры живых игроков на максимум (здоровье, урон, боезапас). Конфигурацию ботов оставить базовой. Скрыть текстовое описание изменений; параметры должны переключаться динамически на глазах у создателя лобби.
    * **Режим №2 (СКОРОСТНОЙ):** Установить максимальный множитель скорости передвижения и действий абсолютно для всех участников матча, включая ботов. Остальные параметры оставить стандартными.
    * **Режим №3 (ПОЛЬЗОВАТЕЛЬСКИЕ):** Действовать как триггер. По нажатию открывать скрытое ранее классическое меню детальной настройки каждого игрока. До активации этой кнопки старое громоздкое меню должно быть полностью скрыто из рендера.

---

### 3. ⚙️ Интеграция настройки чувствительности управления
* **Ответственный исполнитель:** `[Агент: Игровая Механика и Физика]`
* **Задача:** Расширение меню конфигурации игрового процесса индивидуальной настройкой управления.
* **Инструкция для Агента:**
    * Интегрировать переменную `Sensitivity` в общее меню настроек, разместив строку строго *перед выводом версии программы*.
    * Задать диапазон значений от `0` до `10`.
    * Инициализировать базовое значение по умолчанию равным `5` (текущий физический стандарт управления, с которым все ездят). Изменение значения должно линейно масштабировать физику отклика в матче.

---

### 4. 🎁 Система архивации предметов и менеджмент промокодов
* **Ответственный исполнитель:** `[Агент: Бэкенд и Базы Данных]`
* **Суть изменения:** Изоляция удаленных из продажи предметов без физического удаления их сущностей из кода игры.
* **Инструкция для Агента:**
    * Перевести выведенные из магазина предметы в архивный статус (`archived = true`). Полностью скрыть их из каталога внутриигрового магазина (запрет на прямую покупку).
    * Обеспечить полную техническую поддержку и рендеринг этих предметов в матчах для игроков, которые уже ими владеют.
    * В админ-панели разработчика реализовать методы для принудительной выдачи архивных ID конкретным пользователям и генерации промокодов (`промиков`) на их получение.

---

### 5. 🦎 Консервация лут-пула «Chameleon Box»
* **Ответственный исполнитель:** `[Агент: Игровая Экономика и Инвентарь]`
* **Суть изменения:** Перевод кейса в категорию скрытого редкого/коллекционного контента.
* **Инструкция для Агента:**
    * Удалить **Chameleon Box** с витрины внутриигрового магазина.
    * Заморозить и полностью сохранить без изменений внутреннюю структуру кейса: все привязанные скины, имена предметов и наградные медали должны оставаться в лут-пуле.
    * Сохранить рабочим метод `open_box()`. Если бокс присутствует в инвентаре игрока (получен через промокод или админку), механика открытия и выпадения предметов должна отрабатывать штатно.

---

### 6. 🚨 Редизайн системы уведомлений (Фоновый режим и кастомные алерты)
* **Ответственный исполнитель:** `[Агент: UI/UX и Визуальные Эффекты]`
* **Текущая проблема:** Замедление геймплея из-за вызова всплывающих окон Telegram на каждое успешное действие (например, текст «ОТПРАВЛЕНО» при инвайте в лобби).
* **Инструкция для Агента:**
    * Полностью заблокировать вызов нативных окон Telegram при успешном выполнении рутинных действий (приглашение отправлено, лобби создано). Обрабатывать их асинхронно в фоне без алертов.
    * Разработать кастомный мини-виджет для вывода исключительно критических системных ошибок и предупреждений.
    * **Визуальный стиль:** Компактное аккуратное окно, анимированное переливающимся градиентом зеленого цвета с эффектом неонового свечения (*green glow/shimmer effect*). Тесты успешности ("отправлено") внутрь не помещать.
    * **Логика вызовов:** Активация виджета только на негативные сценарии. Примеры строк для вывода:
        * `«Игра заполнена»` — при превышении лимита участников в лобби.
        * `«Некорректный ID»` — при ошибке валидации ID во время поиска игрока/матча.

# БЛОК 5
Техническое задание для GEMINI ANTIGRAVITY: Исправление багов и доработка 2D-шутера
## 1. Интерфейс, уведомления и обработка ошибок (UI/UX)
Важное указание для ANTIGRAVITY: Для выполнения задач из этого блока создай нового изолированного агента, который специализируется на UI/UX, анимациях и жизненном цикле компонентов интерфейса.

Окно кастомных ошибок (Логика видимости и анимация):

Текущая проблема: Окно с ошибкой отображается на экране перманентно.

Решение: Переведи компонент окна ошибок в скрытое состояние по умолчанию. Оно должно монтироваться в DOM / становиться видимым исключительно в момент генерации рантайм-исключения. Анимацию появления сделай плавной, с использованием кривых сглаживания (Spring/Ease) в стиле iOS.

Жизненный цикл и автоскрытие ошибок:

Текущая проблема: Ошибка висит на экране до полной перезагрузки приложения.

Решение: Сделай уведомление об ошибке кратковременным (таймаут отображения 3–5 секунд). По истечении таймера должна срабатывать плавная анимация выхода (fade-out или slide-up), после чего состояние окна сбрасывается.

Коррекция текста валидации режима:

Текущая проблема: При попытке старта без выбранного режима выводится некорректный текст с указанием "слева".

Решение: Измени строку локализации. Полностью удали упоминание расположения ("слева"). Строка должна иметь вид: Сначала выберите режим.

Рефакторинг системных алертов (Замена Telegram UI на кастомный):

Текущая проблема: При клике на игрока в лобби всплывает стандартное нативное окно Telegram (Уверены ли вы, что хотите кикнуть этого игрока из лобби?).

Решение: Полностью откажись от стандартных Telegram-уведомлений. Интегрируй кастомное модальное окно подтверждения, строго соответствующее новому стильному дизайну нашей игры. Примени этот кастомный UI-шаблон для всех диалоговых окон в приложении.

## 2. Механика управления и инпут (Controls)
Важное указание для ANTIGRAVITY: Для выполнения задач из этого блока создай нового изолированного агента, специализирующегося на обработке ввода (Input Management) и физике перемещения персонажа.

Переработка логики чувствительности (Sensitivity):

Текущая проблема: Ползунок чувствительности в настройках ошибочно масштабирует линейную скорость бега персонажа.

Решение: Раздели переменные. Линейная скорость перемещения (movement speed) должна быть константной. Параметр Sensitivity должен влиять исключительно на угловую скорость поворота (angular rotation speed) вектора направления персонажа при круговом движении виртуального джойстика.

## 3. Сетевой код, лобби и мультиплеер (Networking & Lobby State)
Важное указание для ANTIGRAVITY: Для выполнения задач из этого блока создай нового изолированного агента, который разбирается в сетевых протоколах, синхронизации состояний (State Management) и комнатах (Rooms/Lobbies).

Коррекция точек спавна в режиме «Дуэль 1 на 1»:

Текущая проблема: При игре с реальным пользователем (не с ботом) оба игрока возрождаются в одной и той же координате (на одном спавне).

Решение: Настроить жесткое разделение точек респавна для мультиплеера. Хост и подключившийся игрок должны гарантированно появляться на противоположных концах карты, определяясь как враждебные сущности.

Синхронизация состояния лобби после матча и сохранение прав хоста:

Текущая проблема: При выходе из матча обратно в созданное лобби игра повторно запускается только у хоста, а у второго игрока сессия зависает в интерфейсе лобби. Также сбрасываются или передаются права создателя комнаты.

Решение:

Наладь корректный броадкаст состояния готовности (lobby state) для всех участников после завершения сессии, чтобы сессия обновлялась синхронно.

Зафиксируй права хоста (Host Privileges) за создателем комнаты на протяжении всего игрового цикла. При переходе «Лобби -> Матч -> Лобби» право запуска игры и управления комнатой должно бесшовно оставаться у изначального хоста, как будто он и не покидал лобби на время игры.

## 4. Баланс и левел-дизайн (Level Design & Balance)
Важное указание для ANTIGRAVITY: Для выполнения задач из этого блока создай нового изолированного агента, отвечающего за конфигурацию игрового баланса, параметры сущностей (Entity Specs) и геометрию уровней.

Масштабирование карты для «Командного боя» (TDM):

Текущая проблема: Маленькая площадь карты приводит к спавн-кемпингу (игрока убивают мгновенно при возрождении).

Решение: Увеличь общую геометрию (площадь) карты для режима «Командный бой» примерно в 3 раза. Максимально разведи координаты баз возрождения команд в противоположные стороны, чтобы обеспечить безопасную зону спавна.

Коррекция характеристик вражеского ИИ:

Текущая проблема: Избыточное количество здоровья у ботов затягивает игровой процесс в командном режиме.

Решение: В конфигурации режима «Командный бой» уменьши базовый показатель здоровья (HP) для вражеских ботов (ИИ) ровно в 2 раза (на 50%).

# БЛОК 6
ТЕХНИЧЕСКИЙ ПЛАН ОБНОВЛЕНИЯ (2D ШУТЕР В ТГ-БОТЕ)
🛠 ТЕМА 1: Логика режимов и исправление багов спавна
Инструкция для Antigravity: Для выполнения задач этой темы создай нового изолированного агента, который займётся рефакторингом логики состояний комнат и точек спавна.

1.1. Сброс кастомного шаблона при смене режима
Текущая проблема: Если игрок сыграл в режим «Командный Бой» (КБ) с установленным шаблоном «Победительный», а затем переключил режим на «1х1», шаблон не сбрасывается автоматически.

Задача: Реализовать триггер смены режима. При переключении активного игрового режима с КБ на 1х1 (или любой другой), текущий шаблон оформления/настроек должен принудительно сбрасываться на стандартный.

1.2. Фиксация точек спавна в дуэлях
Текущая проблема: В дуэльных режимах игроки возрождаются в случайных местах карты, что ломает баланс.

Задача: Изменить алгоритм спавна для режимов «1х1» и «2х2». Игроки должны гарантированно появляться исключительно на остатках (территории) своей законной базы, а не в случайных точках карты.

💨 ТЕМА 2: Механика дымовых завес (смоков) и генерация карты
Инструкция для Antigravity: Для реализации физики смоков и процедурной генерации создай нового агента. Ему потребуется поработать с графическим отображением (если применимо в 2D) и движком генерации уровней.

2.1. Градиентная невидимость в дыму
Задача: Реализовать механику плавного скрытия игрока. Когда персонаж заходит в зону действия смока, его прозрачность (alpha) должна меняться постепенно — от краев области к центру. Чем ближе к центру смока, тем более невидимым (вплоть до 0%) становится игрок.

2.2. Процедурная генерация объектов (Стены и Смоки)
Задача: Написать или доработать алгоритм генерации карты перед началом матча. Каждый новый раунд/матч дымовые завесы (смоки) и стены должны генерироваться в абсолютно случайных местах по всей площади игровой карты, создавая уникальную геометрию для каждой игры.

2.3. Исправление бага с невидимостью в КБ
Текущая проблема: В дуэлях смоки работают корректно (игрок становится прозрачным). Однако в режиме КБ механика сломана: внутри смока игрок остаётся на 100% видимым, но при этом на карте появляются «невидимые багнутые зоны» (где смока визуально нет, но игрок мгновенно исчезает).

Задача: Провести аудит обработки координат смоков в режиме КБ. Синхронизировать коллизии и визуальное отображение дыма. Убрать фантомные зоны невидимости и вернуть корректное скрытие внутри реального радиуса смока.

⚙️ ТЕМА 3: Панель разработчика и автоматизация версионирования
Инструкция для Antigravity: Для интеграции ИИ и апдейта админки создай нового агента, специализирующегося на логике панели управления, работе с промптами для ИИ и конфигурационными файлами.

3.1. ИИ-генерация патчноутов и авто-инкремент версии
Задача:

🪓 Тема 1: Модернизация режима «Командный Бой 5х5» (TDM)
Инструкция для системы: Создай нового агента для проектирования и реализации этой задачи.

Основная концепция: Динамичный командный матч на очки с бесконечным возрождением, основанный на механике Team Deathmatch из Standoff 2.

Условие завершения игры: Ограничение строго по времени. Матч длится ровно 3 минуты (180 секунд). По истечении таймера побеждает команда, набравшая больше очков.

Система начисления очков: * За каждое убийство игрока противника команде начисляется 1 очко (1 килл = 1 очко).

В верхней панели интерфейса должен отображаться общий счетчик очков обеих команд (например,  (синим) 15 |(оранжевым) 12), а не индивидуальные убийства.

Механика возрождения (Респавн): * При смерти игрок не выбывает из игры.

Через фиксированное время (например, 1-2 секунды) игрок автоматически возрождается на точке спавна своей команды с полным здоровьем.

Дополнительно: Предусмотреть временную неуязвимость (2-3 секунды) после возрождения для предотвращения зажима на спавне (spawn kill).

⚔️ Тема 2: Изменение системы режима «Дуэль 2х2»
Инструкция для системы: Создай нового агента для проектирования и реализации этой задачи.

Основная концепция: Тактический раундовый режим для двух команд по два человека, где цена ошибки высока. Механика полностью опирается на соревновательный режим Standoff 2 (без учета закладки бомбы, упор на ликвидацию).

Условие победы: Игра идет до тех пор, пока одна из команд не выиграет 8 раундов.

Изменение интерфейса: Полностью убрать счетчик убийств (киллов) из верхней панели. Вместо него внедрить счетчик выигранных раундов (например, (синим) 5 | (оранжевым) 3).

Логика работы раундов:

Каждый раунд игроки имеют всего одну жизнь. Механика возрождения (респавна) в течение раунда отключена.

Раунд считается выигранным, если одна команда полностью ликвидировала всех участников противоположной команды.

После завершения раунда счетчик выигравшей команды увеличивается на 1, карта очищается, а все выжившие и погибшие игроки снова спавнятся на своих базах с полным HP для начала следующего раунда.

🎯 Тема 3: Синхронизация режима «Дуэль 1х1»
Инструкция для системы: Создай нового агента для проектирования и реализации этой задачи.

Основная концепция: Классическая дуэль один на один, которая полностью копирует архитектуру и правила режима «Дуэль 2х2», но масштабируется под двух игроков.

Логика и правила:

Использовать абсолютно тот же программный код и раундовую систему, что и в режиме 2х2 (Тема 2).

Игра ведется до 8 победных раундов одного из участников.

Один раунд — одна жизнь. Убил соперника — получил очко в счетчик раундов, после чего происходит перезапуск раунда и спавн на исходных позициях.

В интерфейсе сверху отображается только счет раундов (например, (синим) 7 | (оранжевым) 6).
